================================================================================
                        КЛЮЧЕВЫЕ ОСОБЕННОСТИ КОДА
================================================================================

1. АРХИТЕКТУРА 3-СЛОЙНАЯ
   Layer 1: FETCH (Crawl4AI + fallback requests)
   Layer 2: EXTRACTION (Regex + validation)
   Layer 3: TRAVERSAL (BFS queue management)
   → Независимые слои! Если EXTRACTION падает, TRAVERSAL работает.


2. BFS TRAVERSAL с ПРИОРИТИЗАЦИЕЙ
   Priority 1: /contact, /contacts (forced + prioritized)
   Priority 2: /about, /team (forced + prioritized)
   Priority 3: Остальные ссылки
   → Всегда проверяются важные страницы первыми.


3. MULTI-PASS EXTRACTION
   Pass 1: tel: links (HIGHEST PRIORITY)
   Pass 2: markdown
   Pass 3: cleaned_content
   Pass 4: cleaned_html
   Pass 5: raw html
   Pass 6: tables
   → Несколько источников для максимума контактов.


4. HTML ENTITY NORMALIZATION
   &nbsp; → ' '
   &ndash; → '-'
   &middot; → '-'
   &#160; → ' '
   Решает 35-40% потери контактов!


5. PHONE VALIDATION (STRICT)
   10-11 digits только
   Starts with 7 или 8 (Russian)
   Очищение extensions (доб., ext.)
   Фильтрация noise


6. EMAIL DEDUP ENABLED, PHONE DEDUP DISABLED
   Email: Unique в all_emails dict
   Phone: Все номера добавляются (может быть дублей в результате)
   → Важно! Поведение отличается от интуиции.


7. FALLBACK CRAWLER
   Если Crawl4AI не смог получить первую страницу
   Используется requests.get() вместо AsyncWebCrawler
   Лимит: 5 страниц (vs 10 в main)
   → Graceful degradation при блокировке.


8. RESULTS SLICED
   emails[:10]
   phones[:10]
   sources[:10]
   → Максимум 10 каждого, даже если найдено больше.
