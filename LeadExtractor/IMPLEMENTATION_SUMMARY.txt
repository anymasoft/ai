🎯 АРХИТЕКТУРНЫЕ РЕШЕНИЯ
================================================================================

РЕШЕНИЕ 1: Объединить все источники в один текст
Причина: Максимизировать recall поиска
Результат: Не пропускаем контакты из разных источников
Альтернатива: Обработка по отдельности (v3.0 подход)

РЕШЕНИЕ 2: _normalize_text() перед extraction
Причина: HTML entities и zero-width chars ломают regex
Результат: +35-40% recovery разорванных номеров
Cost: +60 строк кода, O(n) preprocessing

РЕШЕНИЕ 3: 5-pass extraction (tel/mailto/regex/wide/tables)
Причина: Разные источники требуют разного подхода
Результат: Ловим контакты из HTML, text, CSS, tables
Cost: Более сложная логика, но четкая структура

РЕШЕНИЕ 4: RECALL-FIRST валидация (7-15 digits)
Причина: Максимизировать recall, фильтровать позже
Результат: Принимаем ~35% мусора, но находим 95% реальных
Cost: Более грязные данные, требует ML scoring позже

РЕШЕНИЕ 5: NO limit на результаты [:10]
Причина: Frontend должен видеть все найденные контакты
Результат: Больше информации для пользователя
Cost: UI нужно обновить (pagination/virtualization)
