## 🔄 ПОТОКИ ДАННЫХ: БЫЛО VS СТАЛО

### БЫЛО (Проблемная архитектура)

```
АДМИНИСТРАТОР:
  AdminPanel.tsx → GET /api/models/all → AiModel коллекция
  [Видит 16 моделей для выбора]

ПОЛЬЗОВАТЕЛЬ (селектор):
  startupConfig → librechat.yaml modelSpecs.list (HARDCODED)
  [Видит ВСЕ 16 моделей, без фильтрации]

ОТПРАВКА СООБЩЕНИЯ:
  POST /api/conversations { model: 'gpt-4o' }
  → checkSubscription middleware
  → Проверка Plan.allowedModels
  → Если модели нет в плане → 403 ОШИБКА ❌

ПРОБЛЕМЫ:
❌ Дублирование: 2 независимых списка моделей
❌ Рассинхрон: обновление в одном месте требует обновления в другом
❌ Плохой UX: 403 ошибка после выбора модели
❌ Нет фильтрации на UI: селектор показывает недоступные модели
```

---

### СТАЛО (Новая архитектура SSOT)

```
АДМИНИСТРАТОР:
  AdminPanel.tsx → GET /api/models/all → AiModel коллекция
  [Видит 16 моделей для выбора]
  [Выбирает подмножество для каждого плана]
  [PATCH /api/admin/mvp/plans/:planId]
  [Инвалидирует кэш]

ПОЛЬЗОВАТЕЛЬ (селектор):
  ModelSelectorContext → useQuery(['allowedModels'])
  → GET /api/models/allowed (с auth)
  → Backend:
      - Получает userId
      - Находит Subscription
      - Получает Plan.allowedModels
      - Фильтрует AiModel по allowedModels
      - Возвращает только разрешенные модели
  [Видит ТОЛЬКО разрешенные модели] ✅

ОТПРАВКА СООБЩЕНИЯ:
  POST /api/conversations { model: 'gpt-4o' }
  → Модель гарантированно в плане (показана в селекторе)
  → checkSubscription middleware (вторичная защита)
  → ✅ Сообщение отправляется успешно

ПРЕИМУЩЕСТВА:
✅ Один источник истины: AiModel (MongoDB)
✅ Синхронизировано: изменение один раз
✅ Отличный UX: нет 403 ошибок
✅ Фильтрация на UI: селектор показывает только доступные
✅ Кэширование: 60 сек стратегия
```

## 📚 ДОКУМЕНТАЦИЯ ДЛЯ РАЗРАБОТЧИКОВ

### Как расширить систему

**Добавить новую модель:**

1. Отредактировать `/api/models/AiModel.js`:
   ```javascript
   const SEED_DEFAULTS = [
     // ... существующие модели
     {
       modelId: 'new-model-name',
       provider: 'provider-name',
       endpointKey: 'endpointName',
       displayName: 'Human Readable Name',
     },
   ];
   ```

2. Обновить `/librechat.yaml` (опционально):
   ```yaml
   modelSpecs:
     list:
       - name: "new-model-name"
         label: "Human Readable Name"
         preset:
           endpoint: "endpointName"
           model: "new-model-name"
   ```

3. В админке выбрать эту модель для тариф'ов

4. Пользователи получат доступ при обновлении страницы

**Как изменить разрешенные модели для тариф'а:**

1. Открыть `/admin/settings`
2. Перейти на вкладку "Settings"
3. Найти тариф (free, pro, business)
4. Выбрать чекбоксы с нужными моделями
5. Нажать "Сохранить"
6. Кэш инвалидируется автоматически
7. Пользователи получат новые модели при обновлении страницы

## 📞 TROUBLESHOOTING

### Проблема: Селектор показывает только 1-2 модели

**Причина:** Возможно, план пользователя имеет restrictive allowedModels

**Решение:**
1. Проверить `Subscription.plan` пользователя
2. Проверить `Plan.allowedModels` для этого плана
3. В админке добавить модели в план
4. Пользователь обновляет страницу

### Проблема: /api/models/allowed возвращает ошибку 401

**Причина:** Пользователь не авторизован

**Решение:**
1. Убедиться что пользователь залогинен
2. Проверить что токен валидный (не истек)
3. Очистить localStorage: `localStorage.clear()`
4. Перезагрузить страницу

### Проблема: NetworkError при GET /api/models/allowed

**Причина:** Backend не отвечает или network проблема

**Решение:**
1. Fallback автоматически активируется
2. ModelSelector использует startupConfig.modelSpecs
3. Пользователь видит модели, но без фильтрации по плану
4. При восстановлении сети, кэш обновляется

### Проблема: 403 ошибка при отправке сообщения

**Причина:** Модель не в Plan.allowedModels (непредвиденный случай)

**Решение:**
1. Это защитный слой (checkSubscription)
2. Проверить логи: `[PLAN MODEL CHECK]` message
3. В админке добавить модель в план пользователя
4. Пользователь пробует снова
